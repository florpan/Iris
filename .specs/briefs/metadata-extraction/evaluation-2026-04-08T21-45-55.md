## Functional

- [x] Scanning discovers all images in source folder hierarchy — verified: `walkDirectory()` in `scanner.ts` recursively walks directory trees via `fs.readdirSync` with `withFileTypes: true`, filtering by `ALL_SUPPORTED_EXTENSIONS` (11 standard + 14 RAW formats).
- [x] EXIF data extracted (camera model, lens, exposure, ISO, focal length, date) — verified: `extractMetadata()` in `metadata.ts` reads `Make`, `Model`, `LensModel`, `ISO`/`ISOSpeedRatings`, `FNumber`/`ApertureValue`, `ExposureTime`/`ShutterSpeedValue`, `FocalLength`, and `DateTimeOriginal`/`CreateDate`/`DateTime` from exifr output with appropriate fallbacks.
- [x] GPS coordinates extracted when available — verified: `latitude`, `longitude` (via `raw.latitude ?? raw.GPSLatitude`), and `GPSAltitude` extracted with `reviveValues: true` so exifr returns decimal degrees directly.
- [x] IPTC data extracted (title, description, keywords) — verified: `ObjectName`/`title` → `iptcTitle`, `Caption`/`Caption-Abstract`/`description` → `iptcDescription`, `Keywords`/`keywords` → `iptcKeywords` (with multi-format string/array coercion), plus `Copyright`/`Rights` → `iptcCopyright`.
- [x] File metadata stored (size, dimensions, format, modification date) — verified: `stat.size`, `stat.mtime` (stored as `fileModifiedAt`), MIME type from extension map, image dimensions (with RAW-aware field priority), all stored per DB record.
- [x] Camera RAW files (CR2, NEF, ARW, DNG) are supported — verified: 14 RAW extensions in `RAW_EXTENSIONS` sets in both `metadata.ts` and `scanner.ts`; exifr parses TIFF-based RAW containers with RAW-specific dimension extraction logic (`ExifImageWidth`/`ExifImageHeight` preferred over `ImageWidth`/`ImageHeight`).
- [x] Incremental scan skips unchanged files — verified: file size + mtime comparison with 1000ms tolerance in `scanSource()` — skips records where `existing.fileSize === stat.size` and mtime difference is under 1 second.
- [x] Missing files flagged but not deleted from database — verified: after scanning, `missingRecords` are identified by filtering `existingRecords` against `discoveredRelativePaths`, then updated with `missing: true` via `inArray` update. Records are never deleted.
- [x] Scan progress reported via API — verified: `GET /api/sync/status` returns the full `ScanProgress` object from `getScanProgress()`, including `status`, `scanned`, `total`, `added`, `updated`, `skipped`, `missing`, `errors`, `startedAt`, and `completedAt`.

## Compliance

The implementation faithfully follows the spec's technical design decisions:
- Uses `exifr` as specified for metadata extraction
- Stores raw metadata JSON blob alongside structured fields
- Supports all specified image formats (JPEG, PNG, TIFF, WebP, HEIC/HEIF, AVIF, BMP, GIF, and the named RAW formats CR2, CR3, NEF, ARW, ORF, RAF, DNG, RW2)
- Incremental scan tracks both file size and modification time
- Missing files are flagged (`missing=true`) not deleted
- API provides both trigger endpoints (`POST /api/sync/scan`, `POST /api/sync/scan/:id`) and a progress/status endpoint (`GET /api/sync/status`)
- Background async scan with 202 Accepted response matches the spec intent for progress polling

The singleton in-memory scan state is a reasonable tradeoff (noted in the implementation summary) that keeps the progress API fast and avoids schema complexity, consistent with the spec's pragmatic approach.

## Quality

- Clear module separation: `metadata.ts` for extraction, `scanner.ts` for orchestration, `sync.ts` for API
- Defensive coding throughout: all extraction calls wrapped in try/catch with graceful empty-metadata fallback
