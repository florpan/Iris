# Evaluation: Metadata Filtering & Faceted Browse

## Attempt: 1
## Result: PASS

## Functional
- [x] Camera model facet shows all models with image counts — verified: `/api/facets` queries `images.cameraModel` with `count(*)` grouped by camera, guarded by `isNotNull` and `ne("")`; `FacetPanel` renders each `FacetItem` with value + count.
- [x] Lens facet shows all lenses with counts — verified: identical pattern to camera via `images.lensModel`; `FacetPanel` renders Lens section.
- [x] Year/month facet allows date-based drilling — verified: year facets always returned via `EXTRACT(YEAR ...)`; when a year is selected (`dateFrom`/`dateTo` in the same year), month facets are computed and returned nested under the year in the UI with a `pl-3` indent.
- [x] Selecting one facet updates others to reflect the filtered set — verified: `buildBaseConditions` is called once per facet dimension, omitting that dimension's own filter params (`camera: ""`, `lens: ""`, etc.) while including all other active filters. BrowsePage refetches `/api/facets` on every filter change.
- [x] Combined facets produce correct results — verified: `buildBaseConditions` AND-combines all provided filter params into a single WHERE clause; the search endpoint also accepts combined `camera`, `lens`, `dateFrom`, `dateTo`, `format`, `focalLengthMin`, `focalLengthMax`, `isoMin`, `isoMax` params together.
- [x] Stats dashboard shows collection overview — verified: enhanced `/api/stats` returns `totalImages`, `totalSources`, `byFormat` (map of MIME type → count), `dateRange` (earliest/latest), `topCamera`, `totalStorageBytes`; `StatsDashboard` component renders all of these.

## Compliance
The implementation closely follows the spec requirements:
- All six required facet dimensions are present: camera model, lens model, file format, year/month, focal length ranges, ISO ranges.
- Facet values show image counts.
- Facets are combinable and context-sensitive (selecting one updates others).
- Results feed into the thumbnail grid via `BrowsePage`.
- A dedicated stats dashboard is provided.

The architectural decision to create a separate `/browse` page (rather than extending `/search`) is well-reasoned and keeps concerns separated.

## Quality
- Code is clean and consistent with the existing Hono + Drizzle ORM + React patterns in the codebase.
- TypeScript types are well-defined (`FacetValue`, `DateFacetValue`, `RangeFacetValue`, `FacetFilters`, `FacetData`).
- SQL queries use proper ORM methods (`gte`, `lte`, `isNotNull`, `ne`, `sql` tagged template) rather than raw strings.
- The `buildBaseConditions` helper avoids code duplication across all six facet queries.
- Minor: focal length and ISO range queries use `Promise.all` with individual per-bucket queries (N+1 pattern). A single SQL CASE/GROUP BY query would be more efficient, but for typical collection sizes this is acceptable and the code is clearer.
- The 50-value limit on camera and lens facets is documented as a known limitation.

## Issues Found
None blocking. Minor observations:
1. The focal length and ISO bucket counts are computed via N individual queries (one per bucket) rather than a single aggregation query. This is a minor performance concern but not a correctness issue.
2. Month drill-down only activates when both `dateFrom` and `dateTo` are in the same year — single `dateFrom` without `dateTo` does not trigger month breakdown. The variable `yearFilterOnly` is computed but the condition `!params.dateTo && !params.dateTo` is a duplicate check (should be `!params.dateTo && params.dateFrom`). However this is a minor logic issue that doesn't affect the primary month-drill use case (selecting a year from the facets always sets both dateFrom and dateTo).
3. The stats endpoint `totalStorageBytes` uses `::bigint` cast but returns as `Number`, which could lose precision for very large libraries (> 2^53 bytes = ~9 PB). Acceptable for practical use.

None required — all acceptance criteria are met.
