# Evaluation: Map View — GPS-Based Image Display

## Attempt: 1
## Result: PASS

## Functional

- [x] Map displays markers for GPS-tagged images — verified: `MapView` fetches `/api/images/gps` and creates `L.marker([lat, lng])` for each returned GPS image, adding each to the cluster group
- [x] Clustering groups nearby markers at low zoom — verified: `L.markerClusterGroup({ maxClusterRadius: 60, chunkedLoading: true })` wraps all markers; clusters expand on zoom/click via Leaflet.markercluster
- [x] Clicking marker shows thumbnail popup — verified: `marker.bindPopup()` called with HTML containing `<img src="/api/images/${id}/thumb">` (180×120px thumbnail with object-fit cover)
- [x] Popup links to image detail view — verified: popup HTML contains `<a href="/image/${img.id}">View detail →</a>` and the thumbnail itself is also wrapped in a detail link
- [x] Map works with folder, search, and filter results — verified: `FolderPage` passes `sourceId`/`folderPath`; `SearchPage` passes `searchQuery`, `camera`, `lens`, `dateFrom`, `dateTo`, `format`, `minSize`, `maxSize`; backend `/api/images/gps` supports all these filter params
- [x] Map auto-fits to show all markers — verified: after populating the cluster group, calls `map.fitBounds(bounds, { padding: [40, 40], maxZoom: 14 })`
- [x] GPS count indicator shown — verified: `GpsCountBadge` component renders "Showing X of Y images (Z have no GPS)" using `total`, `gpsCount`, `truncated`, and `limit` from the API response; includes "· limit reached" warning when truncated
- [x] Tile source configurable — verified: `mapTileUrl`/`mapTileAttribution` added to `AppConfig` interface; parsed from both YAML config (`mapTileUrl:` key) and env vars (`IRIS_MAP_TILE_URL`, `IRIS_MAP_TILE_ATTRIBUTION`); exposed as `data.map.tileUrl`/`data.map.tileAttribution` via `GET /api/config`; consumed by `useMapConfig` hook with module-level caching; passed from both `FolderPage` and `SearchPage` to `MapView`

## Compliance

The implementation fully complies with spec requirements:

- **Leaflet with OpenStreetMap tiles (configurable)**: Uses Leaflet 1.9.4 with tile URL defaulting to OpenStreetMap; tile source and attribution are both configurable via YAML/env vars as required.
- **Marker clustering**: Leaflet.markercluster 1.5.3 is integrated. Clusters show count and expand on zoom/click.
- **Data source agnostic**: Map is a display mode (not a new data source), toggled from the existing view mode selector in the Header. FolderPage and SearchPage both integrate it cleanly.
- **No GPS = not shown**: The `/api/images/gps` endpoint filters only images where `latitude IS NOT NULL AND longitude IS NOT NULL`. Distinct messaging shown for zero-GPS cases.
- **Count indicator**: Shows "Showing X of Y images (Z have no GPS)" with the exact format described in the spec.
- **Auto-fit bounds**: Uses `map.fitBounds()` with padding as required.
- **Self-hostable tile source**: Tile URL configurable via YAML and env vars with fallback to OpenStreetMap.

## Quality

Code quality is high overall:

- **HTML escaping**: Popup HTML uses a proper `esc()` helper to sanitize filenames before inserting into HTML strings — good security practice.
- **Cancellation**: The GPS fetch effect correctly tracks a `cancelled` flag to prevent state updates after unmount or re-fetch.
- **Vite/Leaflet icon fix**: Standard and well-documented fix for broken Leaflet default icons under Vite asset bundling.
- **Dynamic imports**: Leaflet and markercluster are dynamically imported inside the effect, avoiding SSR/bundle issues.
- **Module-level cache**: `useMapConfig` uses a module-level promise/cache so config is fetched once per session — efficient.
- **Polling for map readiness**: The marker update effect polls `mapRef.current` at 100ms intervals for up to 5 seconds to handle the race between GPS data arrival and Leaflet init. While a polling approach is slightly inelegant, it's well-bounded and documented. A React state flag would be cleaner but the current approach works reliably.
- **`any` refs**: Several Leaflet object refs use `any` type with `@ts-expect-error` / eslint-disable comments — acceptable given Leaflet's imperative API style.
- **SearchPage hidden div**: The existing scroll container is hidden with `className="hidden"` when in map mode rather than conditionally rendered, which is slightly less clean but functional and avoids losing scroll position.

## Issues Found

No blocking issues. Minor observations:
1. The polling approach for map readiness (`tryUpdate` with setTimeout loop) is functional but could alternatively be driven by a `mapReady` state flag for cleaner React patterns — not a defect.
2. SearchPage renders `MapView` without filters when `!hasSearched`, showing all GPS images globally. This is consistent with the "data source agnostic" design principle.
3. Leaflet popup navigation uses `href` anchor tags rather than React Router — causes full-page navigations from map popups. Documented as a known limitation; consistent with how detail links work elsewhere.
