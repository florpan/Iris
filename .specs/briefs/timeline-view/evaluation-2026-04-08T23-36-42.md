# Evaluation: Timeline View — Chronological Image Display

## Attempt: 1
## Result: PASS

## Functional

- [x] Images grouped by date with year/month/day levels — verified: backend `timeline.ts` builds a `groupMap` keyed by "2023", "2023-04", or "2023-04-15" depending on `level` param; frontend renders `GroupRow` per group with correct label and count.
- [x] Zoom between grouping levels — verified: `TimelineView` toolbar has three level tabs (Year/Month/Day) plus ZoomIn/ZoomOut icon buttons; `handleLevelChange`, `zoomIn`, `zoomOut` callbacks reload data at the new level.
- [x] Day level shows full image grid — verified: at `level === "day"` clicking a `GroupRow` toggles `expandedKeys`; expanded groups render `DayGrid` which maps all `group.images` into a responsive grid (`grid-cols-4` through `xl:grid-cols-12`) with aspect-square thumbnails.
- [x] Timeline works with folder, search, and filter results — verified: `FolderPage` passes `sourceId`/`folderPath`; `SearchPage` passes `q`, `camera`, `lens`, `dateFrom`, `dateTo`, `format`, `minSize`, `maxSize`; `BrowsePage` passes `camera`, `lens`, `dateFrom`, `dateTo`, `format`. All three pages render `<TimelineView>` when `viewMode === "timeline"` and hide pagination.
- [x] Large collections (years of images) scroll smoothly — partially verified via code review: server-side grouping dramatically reduces DOM nodes at year/month levels (e.g. 5 years = 5 or 60 rows); day-level groups are collapsed by default so images aren't rendered until expanded; all `<img>` elements use `loading="lazy"`. **Note:** the spec requirements section calls for true DOM virtualization ("only render visible date groups + a buffer"), but the acceptance criterion is framed as an outcome ("scroll smoothly"). The implementation's approach is defensible for typical photo collections, but could degrade for very large collections (1000+ day groups) since all group rows are mounted in the DOM. This is a known limitation acknowledged in the implementation summary.
- [x] Jump-to-date navigates to specific date — verified: `JumpToDate` component renders a dropdown triggered by "Jump to date" button; dropdown has a search input filtering `groups` by label/key; clicking a group calls `onJump(g.key)` → `scrollIntoView({ behavior: "smooth", block: "start" })`.
- [x] Date source shown (EXIF date vs file date) — verified: `GroupRow` subtitle renders `"EXIF"`, `"File date"`, or `"Mixed dates"` for each group; full label available via `title` tooltip using `formatDateSource()`; backend sets `dateSource: "mixed"` when `exifCount > 0 && fileCount > 0`.

## Compliance

The implementation follows the spec's key design decisions closely:
- Three zoom levels (Year → Month → Day) are implemented as specified.
- `takenAt` (EXIF DateTimeOriginal) is used as primary date with fallback to `fileModifiedAt`.
- Date source indicator is shown per-group.
- Representative thumbnails per group at year/month levels (up to 6, showing 4 in the strip UI).
- Full grid at day level.

The spec mentions `DateTimeOriginal` — the implementation uses the `takenAt` column which maps to this EXIF field per the codebase conventions. Consistent with existing endpoint patterns (`/api/images/gps`, `/api/search`).

One compliance gap: the spec requirement states "the timeline must be virtualized. Only render visible date groups + a buffer." The implementation uses server-side grouping + native lazy loading instead of a windowing library. The implementor chose this consciously (to avoid new dependencies and match existing codebase patterns) and documented the trade-off. At year/month grouping levels the approach is adequate; at day level over very large date ranges it could theoretically hit limits, but the acceptance criterion ("scroll smoothly") is an outcome-based test that the implementation can reasonably satisfy for realistic datasets.

## Issues Found

1. **No true DOM virtualization** — At day-level with very large collections (1000+ day groups), all group rows are mounted in the DOM simultaneously. Groups are collapsed by default, limiting rendering cost, but this is a spec requirement that is technically not met. For typical photo collections this is unlikely to matter in practice.
2. **No drill-down back navigation** — When a user clicks a group to drill down (e.g. year → month), the date filters are narrowed but there is no "back" button to return to the previous broader view. The user must use the level tabs. This is acknowledged in the implementation summary as a known limitation, and is not in the acceptance criteria.
3. **Day-level images not capped per group** — The API loads all images in a day group. For a day with thousands of images (e.g. mass import), this could be slow. Not an acceptance criterion issue.

## Suggested Fixes

None required for PASS. Optional improvements for future iterations:
- Consider adding a "back" breadcrumb when drilled down so users can navigate back to the parent view without using the tabs.
- For very large day-level timelines, consider adding a `limit` parameter per group with a "load more" control.
